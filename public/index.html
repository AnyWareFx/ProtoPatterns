Welcome to ProtoPatterns

Welcome to ProtoPatterns, a simple Resource Oriented Architecture (ROA) stack to power your next web application.

ProtoPatterns Design Goals

The ProtoPatterns project values most highly:

  • Simplicity

    The server requires only Rails and the make_resourceful plugin. A single controller provides the base class for your application's RESTful controllers.

  • Convention

    Taking a page from the Rails 'convention over configuration' philosophy, the client micro-framework also uses a predefined directory structure for XSLT view files. That's right, XSLT files. See the following link for the rationale behind this design decision.

ProtoPatterns Design Decisions

With any project, design decisions must be made based on constraints, philosophy, etc. This section outlines the design decisions made for the ProtoPatterns ROA stack.

  • Ruby on Rails

    Maybe you've heard that Ruby and Rails aren't scalable. Maybe you should see for yourself. The list of high-volume sites running on Rails is impressive. No, your eyes don't deceive you. Twitter is on the list.

    In my experience, Rails has been the shortest distance between conception and deployment.

  • XSLT

    That's right, XSLT. I realize that XML has fallen out of favor with the rise of JSON, but there are a few very good reasons for choosing this representation over JSON and a multitude of others.

    1. Compiled Code

      That's right, compiled code. :-) The XSLT processors built into every modern browser are compiled code - most likely C++. Transforming XML into HTML is a (relatively) simple matter, and the compiled code should easily outperform DOM building script. I said "should" because I haven't taken the time to test this assertion. I suppose JIT compiled script could perform almost as well, but I don't even know if any browser has a JavaScript engine sophisticated enough to tackle this...

    2. Simplified API Server

      Since the server only has to return XML and not HTML, the templates can be processed on the client, which has more than adequate power these days.

      Since the server is a simple API server, the client is responsible for rendering the views, as well as implementing the controllers and the interaction patterns controlling the flow between views.

ProtoPatterns Interactive Tests

The ProtoPatterns Interactive Tests are designed to provide interactive confirmation that the stack is working as advertised. Of course, Automated Tests are mandatory and are in the works for both the API server using Test::Unit and the client using the script.aculo.us Test.Unit framework.

Of course, you can view the client script using Firebug, the Chrome Developer Tools, etc. I plan to also serve up the server code itself, complete with Railscasts-like syntax highlighting.
